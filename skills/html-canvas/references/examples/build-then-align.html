Build, then align

↻ Reposted by Zara Zhang Zach Lloyd · CEO, Warp · @zachlloydtweets
TL;DR — In a pre-agent world, alignment had to come before code because coding was the most expensive step. Now that agents collapse the cost of building, the cheapest path to alignment is a working artifact. Build a testable version first, then align around what people actually experience — not around mockups everyone is interpreting differently.

The thesis

If you're spending a lot of time in meetings, docs, and Slack threads before writing code, you're operating on outdated assumptions about what's expensive. The economics of product development have shifted: shipping a tangible thing is now often cheaper than negotiating a shared mental model of one.

"Alignment should come after we have solutions in-hand." — Zach Lloyd

Note: by alignment Lloyd means humans aligning with each other on how a product should work — not humans aligning with agents on how to build. Human/agent alignment (his "Spec and Verify" framing) is a separate, complementary discipline.

Why teams aligned up-front in the first place

Pre-agent, the alignment ritual existed for two specific reasons. Both reasons still exist — what's changed is which intervention solves them most cheaply.

01

Build the right thing on the first try

Coding was the long pole. A wrong guess meant weeks of rework, so up-front debate was a rational hedge against expensive mistakes.

02

Avoid stakeholder surprise

"I thought we were building X, but you built Y." These divergences blow up timelines and trust — and pre-build alignment was the only cheap way to surface them.

What changed: the cost curve flipped

Coding has stopped being the dominant cost. When the build step is cheap, the rational move is to stop optimizing the step before it — alignment — and start using build itself as the alignment tool.

Coding was the bottleneck → now iteration is the bottleneck

Once a thing is built, the team is no longer guessing — they're experimenting on the same artifact. That's the key shift. Pre-build alignment is N stakeholders each imagining a slightly different implementation of a mock; post-build alignment is N stakeholders touching the same thing.

The new design process

Lloyd's old process at Warp had nine steps. Most stand. The change is structural: step 4 (align on the best solution) moves to after step 6 (review the built thing).

01
Clearly state the problem and user goals
Keep · pre-build
02
Explore the solution space (product jam)
Keep · pre-build
03
Mock up different possible solutions
Keep · best-guess only
04
Align on what we think is the best solution
Move later
05
Code the solution
Coding bottleneck — going away
06
Review code & product against the spec
Keep
07
Dogfood and refine internally
Keep · now where alignment happens
08
Bug bash and ensure quality
Keep
09
Ship, monitor, adjust on user feedback
Keep
★
The change isn't "skip alignment." It's "stop bikeshedding solutions before they exist." You still do problem framing, the product jam, and a best-guess design. You just don't spend cycles debating which mock is ideal — you build a few, try them, then align.

Warp's case studies

2023 · Warp Drive · what not to do

Countless hours debating mocks, Slack, PRDs — before any code

Warp's collaborative platform was specced exhaustively up-front. Lloyd's retrospective: "I would not do this again." The pre-build debate didn't produce a better product; it produced a more expensive process.

Recent · Coding-agent suite · what to do

Align on the problem, ship best-guess designs, iterate together

Team aligned on the problem ("make Warp the best place to build with any coding agent"), did not bikeshed solutions, let designers cook starting designs, and built. Alignment emerged through iteration on the real artifact. By launch, the team agreed the solution was strong — without a pre-build alignment marathon.

Takeaways for your own org

  • Audit your pre-build process. If alignment-before-code is eating more time than alignment-after-code, the cost model is inverted for the new economics.
  • Keep problem framing; cut solution debate. Lock the user problem and goals up-front. Don't lock the solution shape — that's what the build will reveal.
  • Build several versions when in doubt. Multiple cheap builds beat one expensive consensus-driven build, because the team experiments on artifacts instead of imaginations.
  • Get it in front of users early. If users can touch it, do that. The cheapest alignment in the world is a user reaction.
  • Don't conflate this with human/agent alignment. Speccing for an agent (clear contracts, verification) is still essential. The shift is in human-to-human alignment about which solution to build.

"This is also a lot more fun." — the closing line is worth taking seriously: process changes that make work faster and more enjoyable tend to stick; ones that only make it faster don't.
